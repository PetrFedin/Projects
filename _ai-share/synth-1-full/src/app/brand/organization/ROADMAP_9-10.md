# Центр управления: план до 9–10/10

Текущая оценка ~7/10. Ниже — что добавить, чтобы выйти на 9–10.

---

## 1. Блок «Требует внимания» (обязательно)

**Сейчас:** алерты есть в `useAttentionAlerts` (сертификаты, незаполненный профиль, задачи, системы), но в интерфейсе нет отдельной панели.

**Сделать:**
- Секция **«Требует внимания»** над или рядом с «Индекс здоровья».
- 4 блока по данным алертов:
  - **Истекающие сертификаты** — список (название, дней осталось), кнопка «Детально» → `/brand` или раздел сертификатов, при необходимости «Устранено» (dismissCertificate).
  - **Незаполненные данные** — список полей (Brand DNA, Keywords и т.д.), «Детально» → `/brand`, «Устранено» (dismissProfile).
  - **Задачи без исполнителя** — список задач, «Детально» → `/brand/team?tab=tasks`, «Устранено» (dismissTask).
  - **Системы в норме** — один зелёный блок, если `integrationIssues.length === 0`; иначе список сбоев и «Детально» → `/brand/integrations`.
- Использовать `alerts`, `getHistory`, `getBlockLabel`, `dismissCertificate`, `dismissProfile`, `dismissTask` из пропсов.
- Ссылки «Детально» вести в разделы по типу алерта (см. `INACTIVE_ALERT_HREF` в page.tsx).

**Результат:** польза раздела резко растёт — видно, что требует действий, и есть переход в нужное место.

---

## 2. Карточки «Разделы организации» — целиком кликабельные

**Сейчас:** клик только по «Открыть раздел» и «Добавить».

**Сделать:**
- Обернуть карточку в `<Link href={card.href}>` или сделать `onClick` → `router.push(card.href)`.
- Внутри оставить кнопки «Открыть раздел» / «Добавить» с `e.stopPropagation()` или как вложенные ссылки с разными `href` (главная — на раздел).
- Добавить `aria-label` для карточки (например, «Перейти в раздел: Профиль бренда»).

**Результат:** быстрая навигация одним кликом по карточке, как на главной.

---

## 3. Ошибка загрузки и «Повторить»

**Сейчас:** при падении загрузки profile/dashboard/health возможны пустой экран или «битые» данные.

**Сделать:**
- В `useOrganizationHealth` различать состояние ошибки (например, `error: Error | null`).
- В блоке «Индекс здоровья» и при использовании профиля в шапке: при `error` показывать короткое сообщение («Не удалось загрузить данные») и кнопку «Повторить» (refetch).
- Опционально: то же для партнёрской экосистемы, когда там появятся запросы к API.

**Результат:** прозрачное поведение при сбоях сети/API.

---

## 4. Возврат с «Изменить» из всех разделов

**Сейчас:** после «Изменить» в активности и сохранения редирект с `?resolved=...` реализован только для `/brand`.

**Сделать:**
- На страницах `/brand/team`, `/brand/integrations`, `/brand/security`, `/brand/subscription`: при наличии `returnResolved` в URL после успешного сохранения/действия делать `router.replace(\`/brand/organization?resolved=${returnResolved}\`)`.
- Логика такая же, как на `/brand`: при монтировании/после сабмита проверять `searchParams.get('returnResolved')` и при сохранении редиректить с `resolved`.

**Результат:** полный цикл «заблокировал → Изменить → сохранил → вернулся, действие разблокировано» для всех типов активности.

---

## 5. Мелкие полировки (для стабильных 9–10)

- **Доступность:** проверить `aria-label` у кнопок периода, календаря, фильтров участников, карточек, попапов.
- **Скелетоны:** при первой загрузке профиля в шапке можно показать скелетон вместо резкого появления текста.
- **Данные с API (когда будут):** подставлять в карточки модулей реальные `stats` (dashboard/API); «24 участника» / «8 онлайн» — из эндпоинта команды/присутствия; активность и алерты — из API вместо моков.
