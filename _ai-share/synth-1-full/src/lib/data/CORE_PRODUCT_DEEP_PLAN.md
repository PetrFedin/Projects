# Глубокий план развития ядра кабинетов: все профили и сквозные задачи

Документ для **владельца продукта** и **заказчика уровня enterprise**: что уже зафиксировано в навигации и доках (`syntha-cores`, `CABINET-INTERACTION-ARCHITECTURE.md`, `CROSS_ROLE_FLOWS.md`, `CORE_OPERATING_CHAIN.md`), а что ещё нужно, чтобы ядро было **работоспособным, связным и оправдывающим платёж** за SaaS.

**Приоритеты:** **P0** — без этого кабинет нельзя честно продавать / демонстрировать end-to-end; **P1** — ожидание крупного клиента через 1–2 квартала; **P2** — углубление и дифференциация.

---

## 1. Как мы оцениваем «качество ядра»

| Измерение | Вопрос заказчика | Текущее состояние (честно) |
|-----------|------------------|----------------------------|
| **Навигация** | За 30 секунд понятно, где моя ежедневная работа? | Уплотнён `syntha-cores`, архив отделён; таблица ролей на `/project-info/roles-matrix` использует **другой словарь имён**, чем подписи групп — это нормально, но требует дисциплины при правках (`role-hub-matrix-table-unification.ts`). |
| **Связность** | Действие в одном кабинете видно в другом (заказ → PO → отгрузка → сообщение)? | Задокументировано в `CROSS_ROLE_FLOWS.md`; **сквозное доказательство в рантайме** — частично (см. §5 там: E2E пробелы). |
| **Данные** | Цифры и статусы из одной модели истины или из моков? | Много экранов на **демо-данных** / JSON; KPI бренд-центра могут подтягиваться из FastAPI, но не везде. |
| **Надёжность** | Сборка и типы зелёные; критические пути не падают? | `tsc` — **0 ошибок** (`npx tsc --noEmit` в корне `synth-1-full`); дальше — удерживать в CI. |
| **Интеграции** | События, outbox, экспорт заказов — живая цепочка? | Jest/Playwright по части B2B/outbox/health — есть; **workflow API `/api/processes/*`** — **файловое хранилище** (`.data/workflow-store.json`, см. `process-workflow-store.ts`): схемы и runtime переживают рестарт на одном инстансе; для serverless — `WORKFLOW_STORE_DISABLED=1`. **E2E API:** `e2e/processes-workflow-api.spec.ts` в `test:e2e:api` (`POST`→`GET`→`PUT`→`GET`). **P1:** БД при нескольких репликах / аудите. |
| **UX** | Не стыдно показать C-level и операционке? | Часть экранов перегружена, часть — заглушки графиков; нужна политика «демо» vs «прод». |

## 2. Сквозные задачи для **всех** профилей (ядро экосистемы)

### P0 — жизненно важно для доверия

1. **`tsc` → 0** и стабильный CI на типах для зоны кабинетов (`app/brand`, `app/shop`, `app/distributor`, `app/factory`, общие `components` ядра). Иначе регрессии неизбежны.
2. **Единый контракт сущностей** в UI: `orderId`, `collectionId`, `poId`, `threadId` — везде, где чаты/календарь/заказы (см. `CROSS_ROLE_FLOWS.md` §3–4). Без этого «связь» остаётся декорацией.
3. **Доменные события / outbox**: не только health-тесты, а **минимум один сквозной сценарий** «действие A → событие → видимость B» по строкам матрицы §2 (хотя бы заказ ↔ производство или заказ ↔ логистика).
4. **Процессы (`/api/processes`)**: для одного узла — файловый store (см. выше); для **кластера / прод с несколькими инстансами** — БД или явная политика «один writer» + отключение записи. Репутационный риск снят для демо/одноинстансного деплоя, остаётся зрелость для enterprise multi-tenant.
5. **Политика демо**: явные бейджи «Демо», отделение от экранов с реальным API (уже начато на отдельных виджетах — распространить правило).

### P1 — ожидание enterprise

1. **Календарь**: единые слои и смысл событий между ролями (уже частично в nav); дальше — **одинаковый payload** событий в API и отображение статусов заказа/PO без рассинхрона.
2. **Чаты**: контекстные треды по заказу/коллекции с RBAC (не в документе — в продуктовой спецификации и тестах).
3. **E2E по приоритетным строкам** таблицы `CROSS_ROLE_FLOWS.md` §5.6 — закрывать по мере готовности, не накапливать «дыры».
4. **Синхронизация имён**: глоссарий «кластер матрицы ↔ подпись сайдбара» в одном месте (уменьшить когнитивную нагрузку на команду и аудиторию `/project-info/roles-matrix`).

### P2 — качество и масштаб

1. Нагрузочные и UX-исследования на **плотности ядра** (не вернуться к «простыне» ссылок).
2. Локализация и консистентность терминов (опт / B2B / PO) по всем пяти кабинетам.
3. Расширенная аналитика и ИИ — только после стабильности P0/P1 на данных не из моков.

## 3. По профилям: что есть, чего не хватает, что убрать/сжать

Ниже — взгляд **«крупная компания платит за этот контур»**.

### 3.1. Бренд

| Угол | Содержание |
|------|------------|
| **Ценность** | Полный контур: продукт → опт → производство → логистика → комплаенс; управление командой и коммуникациями. |
| **Уже хорошо** | Уплотнённое ядро (`Логистика и склад`, календарь со слоями, витрина байера в архиве без шума в `comms`). |
| **Не хватает (P0/P1)** | Связка **заказ B2B → PO → цех** в данных и статусах на одной временной шкале; меньше «магии» KPI без подписи источника; AI-генерация текста/тегов — контракты ответов `json` уже типизируются — добить остальные вызовы. |
| **Избыток / риск** | Очень насыщенный хедер бренд-центра (KPI-карусель): для enterprise нужен режим «спокойный» / скрытие демо-показателей. |
| **Визуал** | Высокая плотность; для C-level — отдельный «обзор» с 3–5 проверенными метриками из API. |
| **Техдолг** | `brand/layout` и тяжёлые страницы production — дожимать `unknown` из JSON/хуков в очередных пакетах `tsc`. |

### 3.2. Магазин (ритейл / закупка)

| Угол | Содержание |
|------|------------|
| **Ценность** | Закупка у брендов, B2B-матрица, склад сети, оплата; календарь с слоями `orders,logistics`. |
| **Уже хорошо** | Уплотнённые B2B-столбцы; таббар B2B согласован с `getB2bHubTabValue` и типом `ShopNavLinkFlat`. |
| **Не хватает** | Сквозной трекинг **мой заказ → статус у бренда** без «прыжков» между кабинетами; единый язык статусов; фильтры маркетплейса (`FilterBarMarketplace`) должны опираться на стабильный API контракт `facets`. |
| **Избыток** | Дублирующие входы в опт из архива — держать под контролем, не размножать. |
| **Заказчик скажет** | «Где один экран „мои открытые заказы и поставки“ без десяти кликов?» — кандидат в P1: сводный дашборд закупки. |

### 3.3. Дистрибутор

| Угол | Содержание |
|------|------------|
| **Ценность** | Те же маршруты опта, что у магазина, но роль сети/опта между брендом и ритейлом; партнёры, логистика, претензии. |
| **Уже хорошо** | Одна колонка партнёров (выставки в подпунктах), один столбец заказов, одна логистика — как задумано для плотности. |
| **Не хватает** | Явная **сценарная связь с магазином/брендом** в UI (не только в доке): кто главный в цепочке при спорной поставке; дашборд «мои сети / мои бренды» на реальных данных. |
| **Избыток** | Проверить архив `DISTRIBUTOR_ARCHIVE_GROUP_ORDER` (дублирующие смыслы «логистика/финансы» не должны путать). |
| **Заказчик скажет** | «Не хочу дублировать Excel — покажите, что уже синхронизировано с брендом». |

### 3.4. Производство (manufacturer / factory)

| Угол | Содержание |
|------|------------|
| **Ценность** | PO, цех, график, отгрузка; без дублирования B2B-реестра в сайдбаре. |
| **Уже хорошо** | Одна колонка «Производство» с подпунктами; партнёры+субподряд; логистика+склад; календарь `tasks,orders,production`. |
| **Не хватает** | Жёсткая связь **номер PO ↔ строка заказа бренда** в интерфейсе и в событиях; QC в архиве — убедиться, что путь для ОТК не «второй сорт». |
| **Избыток** | Любые лишние демо-блоки на экранах цеха — снижают доверие; всё демо — с подписью. |
| **Техдолг** | `production-page-content.tsx` и связанные панели — индексация `unknown`; приоритетная очередь для `tsc`. |

### 3.5. Поставщик (supplier)

| Угол | Содержание |
|------|------------|
| **Ценность** | Материалы, RFQ, VMI, поставка на фабрику; без B2B-витрины магазина. |
| **Уже хорошо** | Один столбец «Материалы и RFQ», логистика+склад; календарь `tasks,orders,logistics`. |
| **Не хватает** | Сквозной **RFQ: бренд создал → поставщик ответил → резерв/отгрузка** с одним id в обоих кабинетах; круговой/ESG блоки — только как демо с явной маркировкой (частично сделано). |
| **Заказчик скажет** | «Где SLA поставки и штрафы за срыв — в одном месте?» — продуктовая фича P1. |

### 3.6. Платформа (Syntha HQ / admin) — кратко

Не «ядро пяти кабинетов», но влияет на доверие: справочники, пользователи, модерация. **P1:** согласованность с кабинетами (роли, организации) без рассинхрона с `useRbac`.

## 4. Ошибки, заглушки и «что сделать в первую очередь» (сводка)

| Категория | Действие |
|-----------|----------|
| **TypeScript** | Дожимать по файлам (`grep "error TS"` → пакеты по 5–10 файлов). **Порядок зон:** **brand** (в т.ч. `brand/merch`, `brand/production`) → общие `components`/`lib` ядра → **`shop/b2b`** → затем **`client/me`** и остальной **`client/`** по необходимости; параллельно **inventory**, если кластер ошибок там. |
| **API процессов** | Файловый store на одном инстансе; **P1** — БД + миграции для горизонтального масштабирования и аудита; при отключении записи — честно в доках/ENV. |
| **Моки** | Инвентаризация: список экранов на `placeholder-data` / статичных JSON → план замены на API или бейдж «Демо». |
| **Тесты** | Закрытие строк матрицы §5.6 `CROSS_ROLE_FLOWS.md` по приоритету бизнеса. |
| **Визуал** | Единый audit: плотность, типографика, пустые состояния («нет данных» с объяснением, а не пустой экран). |

## 5. Предлагаемая последовательность внедрения (квартальный взгляд)

1. **Спринт A:** `tsc` зелёный для модулей кабинетов + исправление критичных `unknown` в production/product.  
2. **Спринт B:** один **сквозной E2E** по выбранной строке матрицы (заказ ↔ второй актор).  
3. **Спринт C:** **workflow API** — файловая персистентность (сделано для single-instance); дальше — **БД / политика** под multi-replica или заморозка обещаний до готовности.  
4. **Спринт D:** продуктовый **«сводный заказ/поставка»** для магазина или дистрибутора (см. боль заказчика выше).  

Детализация задач в таск-трекере должна ссылаться на **id строки** `ROLE_HUB_MATRIX` или **§ таблицы** `CROSS_ROLE_FLOWS.md`, чтобы не терять связь «зачем это ядру».
