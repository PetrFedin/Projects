# Архитектура взаимодействия кабинетов и иерархия навигации

Документ задаёт **кто с кем связан по смыслу**, какие блоки видны в **основном контуре** (`syntha-cores`) сайдбара, а что уходит в **архив** (`archive`) внутри того же профиля. Кодовые якоря: `syntha-nav-clusters.ts`, `brand-navigation.ts`, `shop-navigation-normalized.ts`, `distributor-navigation.ts`, `factory-navigation.ts`.

Сводная **операционная цепочка ядра** (фазы, таблица столбов по профилям, расхождения матрицы и сайдбара, заглушки): **`CORE_OPERATING_CHAIN.md`**. Детальные **цепочки между ролями** (чаты, задачи, заказы, PO, RFQ), чеклист при изменениях и **реестр E2E/Jest / пробелов по рантайму**: **`CROSS_ROLE_FLOWS.md`** (§5). Автопроверки навигации: `npm run validate:cabinet-nav` (матрица ролей, `comms`, ядра `syntha-cores`, **относительные href** по пяти профилям: shop / brand / distributor / manufacturer / supplier — см. `validate:cabinet-nav-hrefs`).

## 1. Модель участников

| Участник        | Роль в сети |
|-----------------|-------------|
| **Бренд**       | Центр: продукт, коллекции, B2B, производство, логистика, партнёры; **взаимодействует со всеми** остальными акторами. |
| **Магазин**     | Ритейл и закупка у брендов; **бренды + дистрибутор** (и связь между ними), без цеха/поставщика материалов как основного контрагента. |
| **Дистрибутор** | Опт между брендом и сетью; **те же границы, что у магазина**: бренды и взаимодействие с магазином/сетью. |
| **Производство**| Цех / исполнение заказов бренда (PO, операции, QC); **бренд + субподряд/другие фабрики**, не розничный закупщик. |
| **Поставщик**   | Материалы, RFQ, логистика поставки сырья; **бренд + другие поставщики/кооперация**, не B2B-витрина магазина. |

Связи в продукте проявляются **только там, где есть общая функция** (заказ, отгрузка, сообщение, документ, маркировка), а не дублированием чужого дерева навигации.

## 2. Общий слой для всех профилей (основной контур)

Во всех кабинетах в контуре закреплены:

- **Команда** — люди, роли, доступы (`/brand/team`, `/shop/staff`, `/distributor/staff`, `/factory/staff`).
- **Связь** — **сообщения** и **календарь** (маршруты профиля).

Бренд дополнительно несёт полный операционный хребет — см. `BRAND_CORE_GROUP_ORDER`.

**Плотность ядра (2026):** в сайдбаре бренда нет дублирующих вторых строк там, где хаб уже ведёт в тот же поток. Пример: **логистика** — одна колонка «Логистика и склад» (хаб + быстрые действия + `hideInSidebar` deep link на склад/материалы/КИЗ); отдельная строка «Склад» убрана из ядра. **Календарь** бренда — без быстрых ссылок на экраны ритейла в ядре: выставки и слоты встреч вынесены в архив **«Витрина байера и площадка»** (`buyer-retail-mirror`), чтобы связь оставалась про процессы бренда, а зеркало байера — про интерфейс магазина.

## 3. Исключения по функциям

| Исключение | Правило |
|------------|---------|
| Разработка коллекции, производственный контур как у бренда | **Нет** в **магазине** и **дистрибуторе**. |
| Группа «Заказы B2B» как у бренда/ритейла | **Нет** группы `b2b` у **производства** и **поставщика**; исполнение — в столбце «Производство» (PO) у фабрики и в столбце «Товар» (`pim`: материалы, RFQ) у поставщика, без дублирующего B2B-реестра в сайдбаре. |

## 4. Основной контур по профилям

Порядок групп: `SHOP_CORE_GROUP_ORDER`, `DISTRIBUTOR_CORE_GROUP_ORDER`, `BRAND_CORE_GROUP_ORDER`, `FACTORY_MFR_CORE_GROUP_ORDER`, `FACTORY_SUP_CORE_GROUP_ORDER`.

### Бренд

`team` → `comms` → `partners` → `development` → `pim` → `b2b` → `production` → `logistics` — остальное в `BRAND_ARCHIVE_GROUP_ORDER`.

### Магазин

`team` → `comms` → `partners` → `pim` → `b2b` → `logistics` — обзор, розница, «Опт: дополнительно» (`shop-b2b-extended`), аналитика, сеть и доступы → архив.

### Дистрибутор

`team` → `comms` → `partners` → `b2b` → `logistics` — обзор, финансы, аналитика, настройки → архив. **Плотность ядра:** одна колонка «Партнёры» (выставки и события — в подразделах, не отдельная строка), один «Заказы B2B» с подпунктами реестра и оформления (режим, создание, быстрый, повтор, черновики), одна колонка «Логистика и претензии» (карта поставок, автопополнение, RMA).

### Производство

`team` → `comms` → `partners` → `production` → `logistics` — материалы, QC, аналитика → архив. **Плотность ядра:** один «Партнёры и сеть» (фабрики + субподряд в подразделах), один «Производство» (`shop-floor`: операции, Гант, выработка, навыки, конфекция, раскрой — в подразделах), одна «Логистика и склад» (хаб + склад и остатки). Календарь в `comms` — со слоями `tasks,orders,production`.

### Поставщик

`team` → `comms` → `partners` → `pim` → `logistics` — комплаенс, аналитика → архив. **Плотность ядра:** один столбец «Материалы и RFQ» (каталог, резерв, RFQ, VMI в подразделах), одна «Логистика и склад». Календарь — `tasks,orders,logistics`.

## 5. Табы и вложенность

Табы и подразделы остаются внутри экранов модулей; сайдбар держит только **столпы профиля**. У бренда скрытые подмаршруты (`hideInSidebar`) сохраняют deep link без лишних уровней меню.

## 6. Эволюция

При новой функции: (1) какие **два актора**; (2) где **первичный** интерфейс; (3) нужна ли **зеркальная точка входа** у второго или достаточно события в связи/заказе.
