# Правила оформления профилей и кабинетов (единый канон)

Документ фиксирует, **как должны выглядеть и устроены** экраны внутри ролевых кабинетов и **профильные** разделы (данные пользователя/организации, настройки, «мой кабинет»), сопоставляет **уже существующие** правила и код проекта и отмечает **пробелы**.

**Аудитория:** продукт, дизайн, разработка, ревью PR.

---

## 1. Назначение и область

| Входит | Не входит (см. отдельные документы) |
|--------|-------------------------------------|
| Оболочка хаба (сайдбар, шапка узла, крошки, основная колонка) | Маркетинговые лендинги вне кабинета |
| Страницы «профиль / я / организация / настройки» внутри кабинета | Полная спецификация BFF/API |
| Вкладки, плотность, токены, иконки, виджеты внутри кабинета | Операционные реестры без профиля — см. `OPERATIONAL_SCREEN_SPEC`, `operational-layout-contract` |

**Профиль** в смысле этого документа: не только `/client/me`, но и **профиль бренда**, карточки организации, настройки узла — всё, где пользователь **настраивает контекст «кто я / моя организация»**, а не ведёт операционный поток заказа целиком.

## 2. Иерархия источников правды

При конфликте приоритет: **токены в коде (`tokens.json` + Tailwind)** → семантические классы (`cabinetSurface`, `registryFeedLayout`) → **этот документ и `CABINET_UI_CANON.md`** → внешний Figma (если расходится — тикет дизайна).

| Источник | Что фиксирует |
|----------|----------------|
| `src/design/tokens.json` + `tailwind.config.ts` | Цвета, отступы, радиусы, шрифты |
| `src/lib/ui/cabinet-surface.ts` + `cabinetHubLayout` | Вкладки кабинета, хром хаба, колонка main, отказ от двойного `px` |
| `src/lib/ui/registry-feed-layout.ts` | Шапки реестров, мягкие панели (`panelCardSoft`) |
| `src/design/CABINET_UI_CANON.md` | Home-хаб, запреты, акценты по ролям (часть формулировок legacy — см. §10) |
| `design-system/synth-1-fashion-os/CABINET_PROFILE_PR_CHECKLIST_RU.md` | Чеклист PR для кабинетов |
| `design-system/synth-1-fashion-os/CABINET_REGISTRY_SHELL_AUDIT.md` | Ожидание `RegistryPageShell` на страницах |
| `src/design/UI_RULES.md` | Примитивы `@/components/design-system`, скелет дашбордов |
| `src/design/JOOR_ORACLE_ENTERPRISE_UI.md` | Целевое качество «плотный B2B / enterprise» |

**Важно:** в `CABINET_UI_CANON.md` §4 упоминаются утилиты в духе `slate-*` как примеры; в текущем коде кабинетов предпочтительны **семантические** классы (`bg-bg-surface`, `text-text-primary`, `border-border-subtle`, `text-accent-primary`). Новые экраны — только семантика + токены.

## 4. Структурный канон хаба (все кабинеты)

1. **Сайдбар слева** — группы навигации из `*NavGroups` (`adminNavGroups`, `brandNavGroups`, `clientNavGroups`, …), единая ширина по `cabinetSidebarLayout`.
2. **Колонка справа** — `CabinetHubMain` (`cabinetSurface.hubMainColumn`): горизонтальный inset **`px-3 lg:px-4`** (у клиента возможен override компактнее — см. `ClientCabinetShell`).
3. **Строка узла** — `CabinetHubTitleRow`: иконка плитки, H1 хаба, бейдж роли (`cabinetRoleLabelRu`).
4. **Полоса секции** — `CabinetHubSectionBar`: крошки, контекст раздела (H2), без второго несогласованного заголовка той же глубины.
5. **`<main>`** — `cabinetHubLayout.mainInner`: дочерние страницы не должны добавлять «лишний» `container mx-auto px-4`; использовать `CabinetPageContent` / `RegistryPageShell` / `PageContainer` по конвенции раздела.

**Логика:** один вертикальный ритм, предсказуемое место для заголовка и действий, без третьей полосы навигации, дублирующей сайдбар (см. запреты в `CABINET_UI_CANON.md` §6).

## 5. Канон страницы «профиля» внутри кабинета

| Элемент | Правило |
|---------|---------|
| Внешняя колонка | Не дублировать горизонтальный padding с хабом: у `/client/me` контент `CabinetPageContent` с **`px-0`**, inset задаёт хаб |
| Макс. ширина текста | Для форм и карточек — `max-w-4xl` в `UserCabinetLayout` (плотный B2B; при табличных дашбордах — `full` / `screen` по смыслу) |
| Шапка страницы | `RegistryPageHeader` (заголовок, lead, действия) в одном стиле с реестрами |
| Вложенные табы | Второй уровень (например «Основные данные / Параметры») — `cabinetSurface.tabsList` + `tabsTrigger` |
| Группы + подвкладки | Как на `/client/me`: верх — `groupTabList` / `groupTabButton*`, низ — `tabsList` + `tabsTrigger` для листовых разделов |
| Состояния | Загрузка / пусто / ошибка — не «голый» текст; минимум карточка с токенами границ |

## 6. Вкладки и плотность

- **Списки вкладок:** `cabinetSurface.tabsList`, **триггеры:** `cabinetSurface.tabsTrigger` (или компактные локальные overrides только в `UserCabinetTabsBar` для клиента).
- **Верхний сегмент групп** (не Radix Tabs): `groupTabList`, `groupTabButton`, `groupTabButtonActive`.
- **Типографика вкладок:** `text-[10px]`, `uppercase`, `tracking-widest` — единый ритм; не смешивать с «sentence case» на соседней полосе без причины.
- **Иконки во вкладках:** Lucide, размер **12px (`size-3`)** в компактном клиентском баре или **14px (`size-3.5`)** в стандартном — не произвольные `h-4`/`h-5` в одном ряду.

## 7. Токены, цвет, иконки, виджеты

| Тема | Правило |
|------|---------|
| Фоны | Страница: `bg-bg-surface` (корневой shell); панели и полосы вкладок: `bg-bg-surface2`; активная вкладка: `bg-bg-surface` |
| Текст | `text-text-primary` / `secondary` / `muted`; не произвольный `text-gray-500` в новом коде |
| Акцент | `text-accent-primary`, `bg-accent-primary` для primary-действий; мягкие подложки `bg-accent-primary/10` |
| Границы | `border-border-subtle` / `border-border-default` |
| Карточки виджетов | `cabinetSurface.panelInner` или `registryFeedLayout.panelCardSoft`; избегать уникальных `shadow-2xl` на рабочих экранах |
| Иконки в виджетах | Предпочтительно **Lucide**, один вес линии; не смешивать с иконками MUI в одной плотной сетке без обёртки |
| KPI / метрики | Компактные ряды; не более **6** KPI на первом экране без UX-ревью (`CABINET_UI_CANON.md`) |

**MUI + Radix:** в проекте есть и то и другое. Для **новых** экранов кабинета предпочтительны **Radix + Tailwind + токены**. MUI — точечно (изолированные виджеты), с выравниванием цветов под семантические токены, иначе визуальный шум.

## 8. Соответствие логике продукта и аналогам (Joor / NuORDER / Oracle)

| Ожидание рынка | Как закреплено у нас |
|----------------|----------------------|
| Плотные рабочие экраны, мало декора | `JOOR_ORACLE_ENTERPRISE_UI.md`, компактность клиентского профиля (`clientMeTabContent`, группировка вкладок) |
| Быстрое сканирование | Иерархия: хаб → секция → таб → контент; ограничение дублирующей навигации (`CABINET_UI_CANON.md` §1–3) |
| Wholesale / showroom качество | Карточки брендов/каталога через design-system; не маркетинговый hero на B2B home |
| Операционный контроль | Операционные экраны — отдельный контракт; профиль не перегружать фиктивными KPI |
| NuORDER-подобный порядок заказов/кабинета | Достигается единообразием вкладок, токенов и плотности, а не копированием чужих пикселей |

**Логика ролей:** переключатель роли (`RolePanel`) не должен дублироваться вторым «переключателем организации» на той же странице без функциональной необходимости (`CABINET_UI_CANON.md` §1.4).

## 9. Роли и удобство

| Роль | Типичный фокус в кабинете | UX-требование к оформлению |
|------|---------------------------|----------------------------|
| Клиент (байер/конечный пользователь) | Заказы, профиль, гардероб | Читаемость, мало жаргона, компактные группы разделов (`/client/me`) |
| Бренд | Каталог, производство, B2B, профиль бренда | Плотные таблицы и формы; длинные страницы — секции и якоря |
| Ритейл (shop) | Заказы, витрина, интеграции | Согласованный хаб с розовым акцентом темы магазина |
| Администратор | Модерация, системные настройки | Нейтральный акцент, предсказуемый хаб |
| Дистрибьютор / фабрика | Цепочки, партии, производство | Emerald/amber акценты в рамках токенов; операционные спеки |
| Маркетинг / демо | Витрины возможностей | Не смешивать с рабочим home B2B без отдельного маршрута |

## 10. Покрытие: что прописано и где пробелы

| Тема | Статус | Комментарий |
|------|--------|---------------|
| Хром `CabinetHub*` | Прописано в коде + `cabinet-surface` | Держать комментарии в `cabinet-surface.ts` актуальными |
| Клиентский профиль `/client/me` | Прописано: модель табов, layout, токены | Расширения — через `client-me-tab-model.ts` |
| Профиль бренда (`/brand/profile` и др.) | Корень: `RegistryPageShell variant="cabinet"` + `RegistryPageHeader`, группы/вкладки — `cabinetSurface` | Внутренние блоки по-прежнему большие — выравнивать секции порциями под §5–7 |
| Единый `RegistryPageShell` | Аудит в `CABINET_REGISTRY_SHELL_AUDIT.md` | Пробелы по `find … page.tsx` — закрывать порциями |
| Светлая/тёмная тема | `next-themes` + `ThemeProvider` | Нужна дизайн-поддержка `dark:` для всех критичных экранов — иначе переключатель ограничен |
| Figma spec pack | Упоминается в чеклисте | Путь к файлу у команды; синхронизация с `tokens.json` при смене палитры |
| Legacy slate в `CABINET_UI_CANON` §4 | Расхождение с семантическими токенами | Считать таблицу §4 **идеей акцента по роли**; реализация — через `accent` и тему магазина/хаба |

## 11. Расширенный чеклист для PR (профили и кабинеты)

В дополнение к `CABINET_PROFILE_PR_CHECKLIST_RU.md`:

- [ ] Нет второго горизонтального `px` у контента внутри хаба без причины (двойной inset).
- [ ] Вкладки — `cabinetSurface` (или согласованный компактный алиас для `/client/me` только).
- [ ] Виджеты на семантических токенах; иконки Lucide в одном размерном ряду с соседними.
- [ ] Тексты интерфейса на русском для пользовательских подписей (аббревиатуры по правилам spec pack).
- [ ] Для изменений home кабинета — критерии `CABINET_UI_CANON.md` §3 и §6.
- [ ] Указаны исключения и ссылка на тикет, если сознательно отступаем от канона.
