# Канон оформления кабинетов (Syntha / fashion B2B)

Документ задаёт **единый визуальный и структурный язык** для кабинетов (`/brand`, `/shop`, `/distributor`, `/factory`, `/admin`, при необходимости `/client`) с учётом специфики: **многоролевость**, **операционные реестры** (заказы, производство, инвентарь), **demo + BFF**, **плотные enterprise-экраны** в духе JOOR/Oracle.

**Связанные источники (не дублировать правила здесь):**

- **`src/design/CABINET_PROFILES_STYLING_RULES_RU.md`** — единый канон **профилей и оформления всех кабинетов** (хром, токены, вкладки, роли, пробелы, Joor/NuORDER).
- `STYLE_GUIDE.md` — типографика, карточки, кнопки, палитра.
- `src/design/UI_RULES.md` — примитивы `@/components/design-system`, скелет страницы.
- `src/design/JOOR_ORACLE_ENTERPRISE_UI.md` — wholesale / planning / analytics.
- `src/design/OPERATIONAL_SCREEN_SPEC.md` + `src/lib/ui/operational-layout-contract.ts` — эталон операционных экранов (таблицы, фильтры, панели).

**Примечание:** таблица акцентов **§4 ниже** задаёт идею по ролям; в коде кабинетов приоритет у **семантических токенов** (`bg-bg-surface`, `text-accent-primary`, …) — см. `CABINET_PROFILES_STYLING_RULES_RU.md` §2 и §7.

---

## 1. Принципы под нашу специфику

1. **Одна главная задача на первом экране кабинета** — «что сделать сегодня» и куда перейти; остальное — второй уровень (сайдбар, разделы).
2. **Одна основная навигация** на home: не комбинировать сайдбар приложения + длинную горизонтальную «ленту всех ссылок» + второй фиксированный рейл с теми же целями без явного разделения ролей.
3. **Данные важнее декора** — никаких крупных маркетинговых hero на **рабочем** home B2B; витрина возможностей — отдельный маршрут или сворачиваемый блок «Демо».
4. **Согласованность ролей** — переключение роли (`RolePanel`) не дублировать вторым «переключателем организации / роли» на той же странице без функциональной необходимости.
5. **Повторяющиеся сущности** (открытые заказы, алерты интеграций) — **один** агрегированный блок на home, не три виджета с пересекающимся смыслом.

---

## 2. Три типа экранов

| Тип                     | Назначение                                          | Обязательный каркас                                                                              |
| ----------------------- | --------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| **Home-хаб**            | Вход в кабинет, обзор дня, быстрые действия         | См. §3 «Паттерн хаба» + примитивы design-system для оболочки                                     |
| **Операционный**        | Списки, фильтры, деталки заказов/артикулов/остатков | `OPERATIONAL_SCREEN_SPEC` + `operationalLayoutContract` + `DataTableContainer` / `FilterToolbar` |
| **Витрина / маркетинг** | Крупные визуалы, storytelling                       | **Не** смешивать с home-хабом; отдельный URL или секция «О продукте»                             |

---

## 3. Паттерн Home-хаба (эталон)

Целевая структура (как на `/shop`, `/distributor`):

1. **Верх: панель дня** — компонент уровня `HubTodayPanel` (или эквивалент с тем же контрактом): KPI из одного источника, **2–4 primary-действия**, компактные алерты.
2. **Низ: карта разделов** — блок «Все разделы» / сетка 2–4 колонок со **ссылками и однострочным описанием**, без дублирования всего сайдбара в виде десятка однотипных карточек.

**Лимиты:**

- Не более **6** KPI в верхней зоне без явного UX-ревью.
- Не более **одного** блока «AI / рекомендации» на home; чат ассистента — либо здесь, либо в отдельной зоне, не дублировать три разных «AI insight» подряд.

---

## 4. База и акценты по кабинетам

**База везде одинаковая** (см. `STYLE_GUIDE.md`):

- Фон и хром: `slate`, панели `bg-white`, границы `border-slate-200/80`, скругление `rounded-xl`.
- Основные кнопки и акцент действий: `slate-900` / нейтраль secondary.

**Акцент роли** — только для ссылок при наведении, бейджей хаба и мелких акцентов (не отдельная «вторая дизайн-система»):

| Кабинет           | Класс акцента (пример)                   | Примечание                        |
| ----------------- | ---------------------------------------- | --------------------------------- |
| Brand             | `text-indigo-600`, hover ссылок в indigo | Уже распространён в brand-области |
| Shop (ритейл-хаб) | `text-rose-600`                          | Как в `HubTodayPanel` shop        |
| Distributor       | `text-amber-600`                         | Как в distributor-хабе            |
| Factory           | `text-emerald-600`                       | Как в factory overview links      |
| Admin             | нейтрально, `slate`                      | Без «яркого» бренда               |

---

## 5. Операционные страницы

- Заголовок страницы: классы из `operationalLayoutContract.pageTitle` или эквивалент из design-system.
- Панели и таблицы: `operationalLayoutContract.panel`, плотные строки, единый `filterBar`.
- Не вводить новые «реестровые» тени и отступы вне `STYLE_GUIDE` / `UI_RULES`.

---

## 6. Запреты на рабочих экранах (home + operational)

- Полноэкранные **маркетинговые баннеры** с декоративным фото и слоганами на **home B2B**.
- **Второй и третий** вертикальный/горизонтальный рейл навигации, дублирующий сайдбар и хаб.
- **Фиктивные KPI** («Активных 14») без привязки к данным или без перехода в список.
- Произвольные `shadow-2xl` и «глянцевые» карточки вне примитивов design-system.
- Три и более **независимых** блоков «рекомендации / AI» на одном скролле home.

---

## 7. Приёмка и миграция

- Новые страницы home кабинета — только по §3–§4.
- Рефактор существующих перегруженных home — поэтапно: сначала убрать дубли навигации и маркетинг, затем выровнять KPI на единый источник.
- Регрессии: затронутые маршруты — смоки / e2e по `tid` и `e2e/README.md`.
