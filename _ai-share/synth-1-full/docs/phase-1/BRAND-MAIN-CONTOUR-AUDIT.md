# Аудит основного контура `brand/` (remove / simplify / keep)

**Область:** навигация из `BRAND_CORE_GROUP_ORDER` и `BRAND_ARCHIVE_GROUP_ORDER` (`syntha-nav-clusters.ts` + `brand-navigation.ts`), страницы под теми же `href`, без перечисления всех 200+ вложенных маршрутов.  
**Канон визуала ядра профиля:** `PROFILE-CORE-SECTIONS-VISUAL.md` (иконка + короткое описание + путь, один сценарий «начало → действие → готово», без дублирования «см. также» и лишних карточек).

Политика: **remove** — убрать с экрана; **simplify** — сократить/структурировать; **keep** — оставить как несущий действие.

---

## Кросс-резка (все кластеры)

| Действие | Что |
|----------|-----|
| **Simplify** | Повторяющиеся вводные абзацы на страницах хабов → одна строка + первичное действие (CTA/таблица/список). |
| **Simplify** | `RelatedModulesBlock` / длинные «см. также» — максимум 1 ряд, только если нет дублирения быстрых действий в сайдбаре. |
| **Simplify** | Многострочные `description` в данных навигации — в UI обрезать до 1 строки или вынести в тултип. |
| **Remove** | Второй «обзорный» слой, если уже есть рабочий список (дублирующий заголовок + тот же список). |
| **Keep** | Demo-контент только с меткой **Демо** / `data-testid` disclaimer, как в `PHASE-1-DEMO-INVENTORY.md`. |

## Основной контур (`syntha-cores`) — по группам

### 1. `team` — Команда (`/brand/team`)

| | |
|---|--|
| **Keep** | Список участников, роли, приглашения; путь в орг-структуру — если клик ведёт к реальному действию. |
| **Simplify** | Любые «онбординг»-блоки > 1 экрана — свернуть в баннер «Завершите настройку» с 1 CTA. |
| **Remove** | Дублирование ссылок на «Профиль/настройки», если те же пути в архиве `brand-admin`. |

### 2. `comms` — Сообщения, календарь

| | |
|---|--|
| **Keep** | Inbox, тред, фильтр по контексту; календарь с слоями `tasks,orders,production` как единая шкала. |
| **Simplify** | Календарь: убрать длинные пояснения «где смотреть ритейл» — 1 ссылка «Ритейл: выставки» в архив или в подпись. |
| **Remove** | Копия объяснений из `description` ссылок навигации в теле страницы. |

### 3. `partners` — Партнёры (ритейлеры; поставщики/фабрики — deep link)

| | |
|---|--|
| **Keep** | Таблица/карточки партнёров, статусы, договор; быстрые действия → поставщики/фабрики без отдельных пунктов в сайдбаре. |
| **Simplify** | Свести вводный текст к: «сеть + статусы»; financial rollup — только если цифры влияют на следующий шаг (кредит, лимит). |
| **Remove** | «Зеркала» и длинные сравнения с кабинетом магазина на этой же странице (это зона `buyer-retail-mirror` в архиве). |

### 4. `development` — Разработка (Workshop2 / `productionWorkshop2`)

| | |
|---|--|
| **Keep** | **Карточка артикула/коллекции** как сущность: шаги от создания до принятого образца; мини-шкала этапов; табы, ведущие к действиям. |
| **Simplify** | `Workshop2DevelopmentIntro` — кратко: разработка+ТЗ → сэмплы; без gate-all-stakeholders в простыне (уже сделано). |
| **Remove** | `Workshop2SeriesOrderHandoffCard` с хаба, если дублирует «серия/B2B» (уже снят с `workshop2/page`). Любые третьи «обзорные» карточки про опт. |

### 5. `pim` — Товары, коллекции, B2B шоурум

| | |
|---|--|
| **Keep** | PIM-список, карточка товара, коллекции как подборка; шоурум как phygital-витрина + лайншиты внутри модуля. |
| **Simplify** | Повтор «матрица в Производстве» — 1 дисклеймер в шапке PIM, не в каждом блоке. |
| **Remove** | Отдельная навигационная строка «лайншиты» в боковом меню (если дублирует `subsections` / шоурум). |

### 6. `b2b` — Заказы B2B

| | |
|---|--|
| **Keep** | Список заказов (operational read-model), деталка, PO, согласование, отгрузка — **цепочка до закрытия**. |
| **Simplify** | Legacy экраны — явная метка **Демо** или выравнивание с `listB2BOrdersForOperationalUi` (см. `PHASE-1-NEXT-ACTIONS.md`). |
| **Remove** | Параллельный `mockB2BOrders` в прикладной логике — через `getB2BOrdersBaseForUi` / read-model. |

### 7. `production` — Производство (цех)

| | |
|---|--|
| **Keep** | Исполнение серий/заказов: снабжение, выпуск, ОТК; матрица SKU — внутри модуля или с deep link. |
| **Simplify** | Не дублировать поля карточки товара из PIM; одна ссылка «карточка в PIM». |
| **Remove** | Kanban «LIVE: от идеи до склада» как обязательный блок на главной цеха (это `production-live` в архиве). |

### 8. `logistics` — Логистика и остатки (хаб)

| | |
|---|--|
| **Keep** | Один хаб: перевозчики, документы, склад ГП, материалы, КИЗ — входы через `quickActions` / subsections, не второй уровень в сайдбаре. |
| **Simplify** | Склад vs материалы: не дублировать одни и те же KPI в двух табах — общий дашборд + фильтр. |
| **Remove** | Второй пункт «склад» в боковом меню (политика уже в комментарии `brand-navigation.ts`). |

## Архив — кратко по кластерам

| Группа | **Keep** | **Simplify** | **Remove** |
|--------|----------|----------------|------------|
| `production-live` | Сценарий LIVE как обучение/карта процесса | Сжать до одного экрана + ссылка в цех | Повтор полного текста этапов цеха |
| `b2b-showcase` | Карта B2B, выставки бренда | Минимум контекста + ссылка в заказы | Длинные сравнения с shop без CTA |
| `buyer-retail-mirror` | Зеркала для сверки UX бренд↔магазин | Группировать ссылки, убрать дубли описаний | Показ всех 10+ ссылок одним полотном без групп |
| `brand-admin` | Профиль, интеграции, документы, настройки, control center | Control center: только плитки с 1 целью | «Полотно» quick links = дубль сайдбара |
| `marketing` | CRM, кампании, медиа — по одному сценарию на экран | quickActions не раздувать в второй сайдбар | Обзорные дашборды без действия |
| `analytics` / `finance` / `tools` | Отчёты с фильтрами и экспорт | Demo label на синтетике | Статика «для красоты» без экспорта/фильтра |

## Как «прогнать» проверку

1. **Навигация:** `npm run validate:cabinet-nav` (href и порядок ядер).  
2. **Ручной проход:** для каждой группы из `BRAND_CORE_GROUP_ORDER` — открыть хаб; проверить: одна роль экрана, нет дублирующего вводного + сайдбар; CTA ведут в следующий шаг.  
3. **Демо:** сверка с `PHASE-1-DEMO-INVENTORY.md` — disclaimer на спорных экранах.
