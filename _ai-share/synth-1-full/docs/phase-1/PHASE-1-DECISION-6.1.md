# Решение §6.1 / §0 — демо и продукт (этап 1)

**Статус:** принято (артефакт репозитория)  
**Дата:** 2026-04-19  
**Связь:** [`CORE_OPERATING_CHAIN.md`](../../src/lib/data/CORE_OPERATING_CHAIN.md) §6.1 · [`PHASE-1-PLAN.md`](./PHASE-1-PLAN.md) §0

Дилемма «демо vs продукт» для первого сквозного усилия **не является взаимоисключающей**. Ниже зафиксирован **двойной контур**: что показываем инвестору/онбордингу, и что считаем **доказуемой связностью** для волны B.

---

## 1. Контур «Демо / онбординг» (сохранить и поддерживать)

| Поле | Значение |
|------|----------|
| **Цель** | Непрерывный рассказ без обязательства нового доменного поведения в каждом PR. |
| **Ось сценария** | Магазин: создание/контур заказа → **platform export** → реестр у бренда; уже покрыто тестами. |
| **Якоря в коде** | `POST /api/b2b/export-order`, `ShopB2bPlatformExportCard`, чеклист **`CORE_OPERATING_CHAIN` §8.7**; Playwright: `b2b-export-order-api`, `b2b-create-order-platform-export-ui`, `b2b-operational-orders-api`. |
| **Обязанность команды** | Не ломать зелёный `npm run test:e2e:api` на этом пути; при UI-правках — сохранять `data-testid` из существующих spec. |
| **Не считается** завершением волны B по «продукту» — это **базовый демо-слой**, не закрытие дыры §5.7. |

---

## 2. Контур «Продукт / связность» (волна B — основная измеримая цель)

| Поле | Значение |
|------|----------|
| **Цель** | Одно доказуемое «действие → зафиксированное состояние → видимость второй стороны» по матрице межролевых потоков. |
| **Строка `CROSS_ROLE_FLOWS` §2** | **B2B заказ (создание → подтверждение)** — бренд ●, магазин ●. |
| **Строка §5.7 (пробел)** | Тот же поток: **«Подтверждение статуса „от бренда“ в UI второй стороны»** — выбран как целевой минимум для закрытия дыры. |
| **Минимально допустимый результат** | Любой из вариантов (выбрать при реализации, не блокирует старт работ): **(a)** E2E: действие в кабинете бренда по смене статуса заказа → `GET` operational order от лица магазина показывает тот же статус; **(b)** контрактный тест API + один assert в UI второй стороны; **(c)** явно зафиксированный в PR «промежуточный» шаг с событием/outbox, если UI второй стороны задерживается. |
| **Не входит в эту волну** | Два браузера/две сессии чата; PO на фабрике; RFQ. |

---

## 3. Сводка для тикетов

**Эпик «Этап 1 / Волна B (продукт)»:** закрыть пробел §5.7 для **B2B заказа** — подтверждение брендом ↔ видимость у магазина.

**Эпик «Этап 1 / Демо-базис»:** удерживать зелёные тесты export + operational orders; полировка UX без смены контракта.

**Идентификаторы:** `wholesaleOrderId`, query чата [`cross-role-entity-ids.ts`](../../src/lib/domain/cross-role-entity-ids.ts), [`glossary-ids.md`](../phase-0/glossary-ids.md).
