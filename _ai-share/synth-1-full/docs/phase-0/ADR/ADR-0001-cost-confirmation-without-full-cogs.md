# ADR-0001: Подтверждение заказа при неполной себестоимости

**Статус:** принято (этап 0)  
**Дата:** 2026-04-19  
**Контекст этапа:** 0 → 2

## Контекст

Себестоимость собирается из BOM, тарифов и ручных полей. Полная автоматическая себестоимость по всем строкам до первого подтверждения заказа не гарантирована. Нужна ясная политика: «только предупреждение» vs «запрет подтверждения».

## Решение

1. **Базовое правило:** переход заказа в статус **`confirmed`** (подтверждение брендом) **запрещён по умолчанию**, если по заказу есть строки с себестоимостью в состоянии **`partial`** или **`manual`** без обязательных полей, заданных политикой бренда (минимум: себестоимость строки или явная пометка «не рассчитано» с причиной).

2. **Исключение (контролируемое подтверждение):** пользователь с ролью **бренд** (или делегат с правом подтверждения заказа) может выполнить действие **«Подтвердить с неполной себестоимостью»**: обязательны **краткая причина** (текст) и запись в **аудите** (кто, когда, какие строки затронуты).

3. **Не считается обходом:** молчаливое подтверждение без аудита — **не допускается** в продуктовом контуре v1.

4. **UI:** до подтверждения показывается **блокирующее предупреждение** с перечислением строк/полей; вторичная кнопка — подтверждение с пунктами (2).

## Альтернативы

| Альтернатива | Почему не выбрана |
|--------------|------------------|
| Только предупреждение | Высокий риск подтверждения заказов с нулевой/завышенной маржой без следа |
| Полный hard block без исключения | Ломает реальные сценарии (срочный заказ, ручная договорённость) |

## Последствия

- Этап 2: в схеме мутаций заказа учитывать флаг **`cogsAcknowledgedAt`** / **`cogsAcknowledgedBy`** при `confirmed`.
