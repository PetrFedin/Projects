# ADR-0002: Частичные отгрузки и приёмка — объём v1 vs v1.1

**Статус:** принято (этап 0)  
**Дата:** 2026-04-19  
**Контекст этапа:** 0 → 2

## Контекст

Частичные отгрузки по строкам заказа и многошаговая приёмка увеличивают сложность UX, складских статусов и документооборота.

## Решение

1. **В v1 основного контура** заказ в исполнении моделируется как **одна логистическая цепочка на заказ** в смысле **клиентского статуса**: допускаются статусы `in_fulfillment` → `shipped` → `delivered` **без** обязательной поддержки **нескольких независимых отгрузок по подмножествам строк** в одном `wholesaleOrderId`.

2. **Отложено на v1.1+:** частичные отгрузки по строкам, несколько `shipmentId` на один заказ с разбиением количеств по строкам, частичная приёмка магазином по строкам с разными датами УПД.

3. **Исключение v1:** если бизнесу нужна фиксация «отгрузили часть» **до v1.1**, используется **временный обход**: статус **`on_hold`** с комментарием и задачей в календаре (не нормализованный учёт по строкам).

## Альтернативы

| Альтернатива | Почему не выбрана |
|--------------|------------------|
| Полные частичные отгрузки в v1 | Риск срыва сроков этапа 2 и раздувания складской модели |
| Полный отказ от частичных навсегда | Нереалистично для fashion B2B — перенос на v1.1 |

## Последствия

- Этап 2: модель данных заказа может **не** включать line-level shipment splits до старта эпика v1.1.
