# ADR-0003: ЭДО и РФ-документы в v1 — поля и статусы vs интеграция

**Статус:** принято (этап 0)  
**Дата:** 2026-04-19  
**Контекст этапа:** 0 → 2

## Контекст

«ЭДО» может означать продуктовые поля и процесс или полноценный обмен с оператором ЭДО. Для этапа 2 нужна фиксация глубины.

## Решение

1. **В v1** реализуется **минимальный продуктовый контур**: хранение и отображение **полей** (реквизиты, номера, даты, статусы согласования), ручной ввод там, где нет интеграции, **статусы** документа в жизненном цикле (черновик / отправлен / принят контрагентом — уровень детализации по согласованию с UX).

2. **Полноценная интеграция** с оператором ЭДО (отправка/получение УПД, подписи, роуминг) — **не входит в обязательный объём v1**; планируется **волной 2** после стабилизации заказа и мастер-данных.

3. **Маркировка (ГИС МТ):** в v1 — **поля** (КИ, статус вводимости) и процессные статусы; **автоматическая выгрузка/проверка** в ГИС МТ — по отдельной дорожной карте (не блокирует этап 2).

4. **НДС / УПД:** обязательный минимум полей — см. [`PHASE-0-RF-FIELDS-MINIMUM.md`](../PHASE-0-RF-FIELDS-MINIMUM.md).

## Альтернативы

| Альтернатива | Почему не выбрана |
|--------------|------------------|
| Сразу интеграция ЭДО в v1 | Непредсказуемые сроки и зависимость от оператора/договоров |
| Никаких полей до интеграции | Ломает демо и ручной контур РФ |

## Последствия

- Этап 2: API и UI заказа содержат **RF-документ** как сущность с полями, без обязательного webhook ЭДО.  
- Юридический sign-off по полям — владелец риска в [`PHASE-0-RISK-REGISTER.md`](../PHASE-0-RISK-REGISTER.md).
