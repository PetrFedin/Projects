# Аудит: Organization overview + профиль бренда vs UI Standard v1

Оценка расхождений с [UI_STANDARD_V1.md](./UI_STANDARD_V1.md). Иерархия экрана — в [ORGANIZATION_OVERVIEW_UI_HIERARCHY.md](./ORGANIZATION_OVERVIEW_UI_HIERARCHY.md). План работ — [ORGANIZATION_OVERVIEW_REFACTOR_BACKLOG.md](./ORGANIZATION_OVERVIEW_REFACTOR_BACKLOG.md).

**Область аудита:** в первую очередь `organization-overview-content.tsx` и страница `/brand/profile` как «кабинет бренда» вне главной витрины.

---

## Таблица несоответствий

| ID | Область | Несоответствие | Желательное действие | Приоритет |
|----|---------|----------------|----------------------|-----------|
| S1 | Profile + Org | Нет `RegistryPageShell` / паттерна `RegistryPageHeader` из стандарта | Ввести общий layout для кабинета бренда (ширина, отступы, зона заголовка) | P1 |
| S2 | Org overview | Два уровня «шапки» (Hub + заголовки `SectionBlock`) + плотная микро-типографика и ALL CAPS на длинных подписях | Выровнять иерархию заголовков и типографику под раздел **Visual Discipline** в UI Standard v1 | P2 |
| S3 | Org overview | Дублирование контроля периода (в Hub header и снова в блоке «Недавняя активность») | Один источник правды периода (props / контекст) и один явный UI-контроль на экран | P2 |
| S4 | Org overview | Внутренний `Card` с рамкой внутри каждого `SectionBlock` при уже выделенном хедере секции | Убрать двойную визуальную рамку или унифицировать через общие токены (`registryFeedLayout` и т.п.) | P2 |
| S5 | Org overview | Многократное повторение одной геометрии мини-карточки (~200×280, футер «Открыть раздел») | Вынести общий компонент карточки + единые отступы/тени | P3 |

## Примечания

- Стандарт ориентирован на **operational/registry** поверхности; длинный **профиль бренда** может оставаться формо-ориентированным, но **оболочка** (shell, ритм отступов) всё равно выиграет от S1.
- Пункты S2–S4 снижают когнитивный шум без обязательной смены бизнес-логики.
