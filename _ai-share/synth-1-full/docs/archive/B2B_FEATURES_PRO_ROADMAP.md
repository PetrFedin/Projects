# B2B фичи: что сделано и что усилить до проф-уровня

Краткий аудит двух добавленных блоков и план доведения до production-ready.

---

## 1. Трекинг заказа и прозрачность поставки

**Сделано:**
- Блок «Отгрузка и доставка» в карточке заказа (магазин и бренд): дата отгрузки, трек-номер, этапы (Собран → Передан в доставку → В пути → Доставлен).
- Баннер «Заказ отгружен» с возможностью скрыть.
- Мок-данные в `src/lib/b2b/order-shipment-tracking.ts` (жёстко зашиты ID заказов B2B-0012, B2B-0011).

**Что усилить до проф-уровня:**

| Направление | Сейчас | Про-уровень |
|-------------|--------|-------------|
| **Источник данных** | Синхронный мок по `orderId` | API `GET /orders/:id/shipment` или JOOR ASN; при недоступности — fallback на мок. |
| **Загрузка и ошибки** | Нет | Состояния loading / error / empty; скелетон в блоке; retry. |
| **Интеграция с перевозчиком** | Ссылка «перевозчик» — тост | Реальная ссылка на трекинг (DHL/UPS/СДЭК по `carrier` + `trackNumber`). |
| **Этапы** | Фиксированный набор | Конфигурируемые этапы из API или по типу доставки. |
| **Уведомления** | Только баннер на странице | Push/email «Заказ отгружен» и «Доставлен» (настройки в профиле). |
| **Тесты** | Нет | Unit: логика трекинга; интеграция: страница заказа с моком. |

---

## 2. Аналитика и отчёты для партнёра («Мои отчёты»)

**Сделано:**
- Страница «Мои отчёты» (`/shop/b2b/reports`): продажи по брендам, топ SKU, sell-through, план/факт закупок.
- Экспорт в CSV по каждому блоку и «Экспорт всех» (4 файла).
- Мок-данные в `src/lib/b2b/partner-reports-data.ts`.
- Ссылка в B2B-хабе.

**Что усилить до проф-уровня:**

| Направление | Сейчас | Про-уровень |
|-------------|--------|-------------|
| **Источник данных** | Статические массивы | API с фильтрами (период, сезон, бренд); fallback на мок при 5xx/offline. |
| **Фильтры** | Нет | Период, сезон, бренд; сохранение в URL или localStorage. |
| **Загрузка и ошибки** | Нет | Скелетоны таблиц, сообщение об ошибке и «Повторить». |
| **CSV** | Разделитель `;`, BOM есть | Экранирование полей с `;` и кавычками (RFC 4180). |
| **Большие объёмы** | Всё в одном запросе | Пагинация; экспорт в фоне с уведомлением о готовности. |
| **Кэш** | Нет | TTL кэша; метка «данные на дату»; кнопка обновления. |
| **Тесты** | Нет | Unit: exportToCsv (формат, экранирование); страница с моком. |

---

## 3. Общие шаги

1. **Слой данных:** Хуки `useOrderShipmentTracking(orderId)` и `usePartnerReports(filters)` с вызовом API и моком при ошибке (по аналогии с `useProductionData`).
2. **Контракты API:** DTO для shipment и отчётов в общих типах; согласование с бэкендом.
3. **Метрики:** Логирование открытия отчётов, экспорта CSV и просмотра трекинга.

---

## 4. Уже внесённые улучшения (задел под про-уровень)

- **CSV (partner-reports-data.ts):** Экранирование полей по RFC 4180: поля с `;`, кавычками или переносами оборачиваются в кавычки, кавычки внутри дублируются. Опция `delimiter` для выбора разделителя.
- **Трекинг (order-shipment-tracking.ts):** Экспортирован хук `useOrderShipmentTracking(orderId)` с сигнатурой `{ data, loading, error, refetch }`. Пока отдаёт мок синхронно; при появлении API достаточно подставить fetch и состояния загрузки/ошибки.
