## 1. Информационная архитектура (высокий уровень)

Один Next-приложение: кабинеты **бренд** и **shop** делят BFF/API и аутентификацию; **общий слой коммуникаций** пересекает роли с разной глубиной UX.

```mermaid
flowchart TB
  subgraph shared["Общий слой (Fashion OS)"]
    NEXT[Next.js: маршруты / UI shell]
    BFF[BFF + REST: b2b, brand API, …]
    TENANT[Tenant / демо-bootstrap]
  end

  subgraph brand["Кабинет бренд"]
    B_PROD[Производство: W2, tech pack пилот]
    B_B2B[Коллекции, B2B operational orders]
    B_COMMS[Messages + Calendar полный]
  end

  subgraph shop["Кабинет shop"]
    S_B2B[Зеркало B2B заказов]
    S_COMMS[Messages + календари B2B/общие]
    S_OTHER[Розница / склад — контекст]
  end

  subgraph adjacent["Смежные роли"]
    FACTORY[Производство: handoff, вкладки чат/календарь]
    SUPPLIER[Поставщик пошива: в данных W2, UX через фабрику]
  end

  brand --> NEXT
  shop --> NEXT
  NEXT --> BFF
  BFF --> TENANT
  brand -->|handoff / сроки| FACTORY
  brand -->|операционный заказ| shop
  B_PROD -.->|контрагенты, план| SUPPLIER
```

## 2. Золотой путь и ветки A / B / C

**Spine:** бренд → точка входа → три опоры по порядку (или короткие «окна» с возвратом). **Ветки:** A = ТЗ→образец, B = коллекция→операционный заказ, C = чат+календарь.

```mermaid
flowchart TD
  START([Старт демо: роль бренд]) --> ENTRY["/brand/profile\nточка входа"]

  ENTRY --> PILLAR{Три столба}
  PILLAR --> A["A: Workshop 2\nТЗ → образец"]
  PILLAR --> B["B: Коллекции\n→ operational order"]
  PILLAR --> C["C: Comms слой\nmessages + calendar"]

  A --> A1["/brand/production/workshop2"]
  A1 --> A2["…/c/SS27/a/demo-ss27-01\n(+ опц. tech pack)"]
  A2 --> MERGE([Слияние в рассказ])

  B --> B1["/brand/collections"]
  B1 --> B2["/brand/b2b-orders → [orderId]"]
  B2 --> B3["коротко: /brand/integrations"]
  B3 --> MERGE

  C --> C1["/brand/messages"]
  C1 --> C2["/brand/calendar"]
  C2 --> C3["опц.: пол производства\nвкладки чат/календарь"]
  C3 --> MERGE

  MERGE --> SHOP{"Нужен shop\n1 мин?"}
  SHOP -->|да| SH["/shop/b2b-orders\nзеркало контракта"]
  SHOP -->|нет| END([Закрытие: смоки + честные подписи])
  SH --> END
```

## 5. Легенда: что «реально», что демо / пилот


| Символ | Значение                                                                                                                  |
| ------ | ------------------------------------------------------------------------------------------------------------------------- |
| **●**  | Реальный маршрут, UI и контрактные смоки в репозитории                                                                    |
| **◐**  | Реальный UI/API, но данные: **read-model**, демо-tenant, seed или **localStorage** досье — озвучить как демо-персистенция |
| **○**  | Пилот / **env-gated** (напр. tech pack: S3, `DATABASE_URL`, preflight) — не продавать как shipped                         |
| **◌**  | Заглушка, empty state, следующий спринт — одна честная фраза                                                              |


На слайде/доске можно дублировать легенду в углу: **● прод-путь · ◐ демо-данные · ○ пилот по env · ◌ не обещаем**.
