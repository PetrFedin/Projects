## 2. Пробелы полноты (чего не хватает *между* четырьмя файлами)

### 2.1 Самодостаточность и устаревшие ссылки

| Файл | Проблема |
|------|----------|
| `FOCUS` (стр. 82) | Отсылка к `INVESTOR_IA_CUTLINE.md` и «детали в spine» — эти документы **вне** канона из четырёх; читатель не обязан их иметь. |
| `GAP` (стр. 5, 50) | Входы: `FUNCTION_SPINE…`, `INVESTOR_IA_CUTLINE`, `VISUAL_INVESTOR_DEMO` — **архив**; факты уже частично в `FINAL`/`GAP`, но **ссылки вводят в заблуждение**. |
| `PLAN` (стр. 4–5, 98) | Опора на `INVESTOR_IA_CUTLINE` и матрицу `UNIFIED_ECOSYSTEM_VERIFICATION.md` — второй путь **вне** `.planning/research/…` канона. |
| `FINAL` (стр. 3, 265–293) | Ссылается на `KEEP_IMPROVE…`, `VISUALIZATION_COMPLETENESS…`, `DIAGRAMS_RETAINED_STRUCTURE_MASTER` — для «только 4 файла» это **дубликаты или архив**. |

**Вывод:** четыре файла **достаточны по содержанию**, но **метаданные и перекрёстные ссылки** надо **привести к канону** (заменить на «см. раздел X в GAP/FINAL» или приложить **одно приложение** в `FINAL`).

### 2.2 Детализация, которой нет ни в одном из четырёх (или только намёк)

| Пробел | Почему важно | Где лучше дописать |
|--------|--------------|-------------------|
| **Единый список URL демо** (фиксированные строки, без «например») | Сейчас размазано: `FOCUS` шаг 1, `PLAN` таблица метрик, `FINAL` таблицы — **нет одной нумерованной последовательности вкладок**. | Приложение **«Демо URL v1»** в `FINAL` или в конец `FOCUS`. |
| **Продуктовые решения (ADR в мини-форме):** два уровня чата; один канонический календарь; thread id ↔ сущность | В `GAP` §6 — вопросы; в `FOCUS` — сознательный non-goal realtime; **ответов «как делаем» нет**. | Новый раздел **«Решения v0.1»** в `GAP` (3–5 буллетов) или отдельный `ADR_FOCUS_*.md` *внутри канона* (тогда пять файлов). |
| **Специфика строки operational order** (размеры, delivery, ship window) — факт из кода | `GAP` пишет «уточнить по DTO» — **детализации нет**. | Таблица **полей v1** в `GAP` после однократного чтения `operational-order-dto` / UI. |
| **Сквозной контракт OO → агрегация → PO** (шаги API, id связей) | Есть **имена** (`OrderAggregate`, `CreatePOFromSamples`), нет **пошагового контракта** для разработки. | Подраздел в `GAP` или `FINAL`: 5–10 строк «событие → handler → сущность». |
| **sequenceDiagram** (handoff, B2B→PO, уведомление→чат) | В `FINAL` только **backlog** визуализации, самих диаграмм **нет**. | Вставить 1–2 mermaid-блока в `FINAL` § после диаграммы 2. |
| **RBAC: кто меняет lifecycle / read-only ритейл** | Только пункт backlog в `FINAL`. | Таблица **роль × действие × да/нет** в `GAP` или `PLAN` Фаза 2. |
| **Полный путь поставщика** (закупка–продажа–склад) vs «через фабрику» | `GAP` честно «частично»; **нигде нет целевого scope** «что входит в v1 снабжения». | Абзац в `FOCUS` «сознательно» или в `GAP` § supply scope. |
| **Скрипт демо для ведущего** (что говорить, порядок кликов, fallback) | Был в архивном `INVESTOR_DEMO_VARIANT…` — **в четырёх нет**. | Приложение в `FINAL` или конец `PLAN` §6. |
| **Связка Python FastAPI с экраном** | `FOCUS` запрещает пустой питч — **положительной связки нет**. | Одна строка в `GAP` или `PLAN`: «какой экран дергает какой сервис» или «вне демо v1». |

## 3. Оценка полноты по «типам детализации»

| Тип детализации | Оценка | Комментарий |
|-----------------|--------|-------------|
| Продуктовая стратегия | **Высокая** | `FOCUS` |
| Соответствие процесс ↔ код ↔ gap | **Высокая** | `GAP` |
| План работ и файлы | **Высокая** | `PLAN` |
| Визуализация и страницы | **Высокая** | `FINAL` |
| Зафиксированные **решения** (не только вопросы) | **Низкая** | Нужно дописать |
| **Единый** чеклист URL + скрипт демо | **Средняя** | Размазано |
| **Контракты данных** (DTO, связи OO→PO) | **Средняя** | Нужен тех. абзац |
| RBAC / исключения / sequence | **Низкая** | Только backlog в `FINAL` |

## 4. Рекомендуемые действия (без обязательности трогать архив)

**Минимум (чтобы «четыре файла» были полной базой):**

1. **Правка ссылок** в `FOCUS`, `GAP`, `PLAN`: убрать зависимость от архивных имён; везде «см. `GAP` §… / `FINAL` §…».
2. **Один раздел в `FINAL`:** «Демо URL v1» + «Скрипт ведущего (10–15 мин)» — перенести смысл из архивного demo-variant.
3. **Один раздел в `GAP`:** «Решения v0.1» (чат: два уровня или сходимость; календарь: канон; thread: order vs collection) — даже если ответ «TBD до спринта», это должно быть **в каноне**.
4. **Один подраздел в `GAP` или `FINAL`:** контракт **eligible коллекции** + **OO→AGG→PO** в 5–10 предложениях после чтения кода (отдельная микро-задача разработчика).

**Опционально (повышает полноту до «спеки»):**

5. В `FINAL` — **sequenceDiagram** ×2 (из backlog).  
6. В `GAP` — таблица **полей** operational order v1.  
7. В `PLAN` Фаза 2 — мини-таблица **RBAC** для spine-действий.

## 5. Итог одной фразой

Четыре документа **достаточны как каркас** (стратегия, разрывы, план, визуализация и страницы); для **полной** детализации исполнения не хватает **зафиксированных продуктовых решений**, **единого демо-чеклиста URL со скриптом**, **явных контрактов OO→PO и gate коллекции**, плюс **очистки ссылок** на не-канонические файлы.
