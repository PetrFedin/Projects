## Что это за продукт

Это **Fashion OS**: кабинеты бренда, ритейла (shop), фабрики и смежных ролей в одном Next-приложении. В центре — **операционная цепочка моды**: от технического задания на артикул и передачи в производство до **оптовых заказов и витрин коллекции**, плюс **общий слой коммуникаций** (сообщения и календарь), чтобы сроки и договорённости не терялись между кабинетами.

**Метрики успеха демо (канон):** один проход spine A→B→C за **≤25 мин** по `**docs/INVESTOR_DEMO_RUNBOOK.md`**; **11** URL максимум (`PLAN` §7.7, п.11 опционален); зелёный CI-поднабор — `PLAN` §6; честные подписи mock / LS / env-gated на W2 и tech pack.

## Три столба фокуса

### (A) Разработка артикула: ТЗ → образец

- Хаб Workshop 2 и рабочее место артикула: досье фазы 1, секции, signoff, версии, merge, lifecycle, события, финальный экспорт.
- Связка с **tech pack** (presign, индекс, handoff к производству) — пилот, зависит от env (S3/DB).
- Справочники **контрагентов пошива** и привязка к плану — read-model через API бренда.
- Демо-контур зафиксирован смоками (`workshop2-smoke`, slug’и вроде `SS27` / `demo-ss27-01`); часть досье может опираться на localStorage в демо.

### (B) Коллекции + B2B: заказы и шоурум

- **Операционные B2B-заказы** — реестр и карточка у бренда и в shop, единый BFF-контур (`/api/b2b/operational-orders`, v1).
- **Коллекции, linesheets, showroom** — витрина сезона для опта; в коде много маршрутов под нарратив JOOR/NuORDER — для фокуса важен **один** сквозной сценарий, а не вся ширина меню.
- **Канон демо-витрины (зафиксировано):** основной вход `**/brand/showroom`** (`ROUTES.brand.showroom`); альтернатива для сценария «linesheet» — `**/brand/b2b/linesheets**`. Далее в сквозном демо — `/brand/collections` → `/brand/b2b-orders` (порядок — `PLAN` §7.2; чеклист и смоки — **§7.7–7.8**; контракты — `GAP` §7).
- Интеграции как «карта экосистемы» и связь с B2B-реестром — опора для демо целостности.
- Коллекции в кабинете бренда (`/brand/collections` и навигация PIM/showroom) — якорь витрины сезона перед узким демо-сценарием в B2B.

### (C) Чат + календарь как сквозной слой

- У бренда и shop — полноформатные **Messages** и **Calendar** с контекстом B2B и навигационным приоритетом «comms».
- На **полу производства** бренда — быстрые вкладки «чат / календарь» (локальный UX поверх production shell; не обязательно тот же маршрут, что `/brand/messages`).
- У фабрики, дистрибьютора, клиента, админа — свои пары `messages` / `calendar` в `ROUTES`; кросс-роль — через согласованные ссылки и сценарии, а не обязательно один realtime-backend в текущем питче.

## Роли в одну строку (в рамках трёх столбов)

- **Бренд** — ведёт досье W2 и tech pack handoff, формирует B2B-витрину и **операционные заказы** с партнёрами, ведёт переговоры и сроки в сообщениях/календаре.
- **Магазин (shop)** — зеркало B2B-заказов и закупочного контекста, розница/склад, сообщения и календари для согласования с брендом.
- **Производство (фабрика)** — принимает следы handoff/W2, смотрит производственный календарь и переписку в своём кабинете вдоль тех же осей «заказ / срок / артефакт».
- **Поставщик (пошив и др.)** — фигурирует в данных W2 (контрагенты, планы); основной UX для исполнителя — через **фабрику/производство** и связанные коммуникации, а не отдельный «четвёртый столб».

## Режим «только три столба» без вырезания кода

- **Код и маршруты вне spine не удаляем** — репозиторий остаётся полным продуктом для внутренних ролей и deep links.
- **Навигация:** в ядре (в т.ч. investor spine по **`PLAN` §8**) показываем только то, что относится к **A / B / C**; остальное — **вторичные кластеры**, **More**, при необходимости подпись **«вне фокуса»** / демо.
- **Демо и приёмка:** в runbook и зелёный **CI-поднабор** входит только согласованный список URL (**`PLAN` §7.7–7.8**, **`PLAN` §6**); хвост **не** тянем в инвесторский проход без отдельного решения.
- Детали реализации навигации и флагов — **`PLAN` §4**, **§8**, **`FINAL`** (IA, диаграмма 3).

## Что сознательно не делаем сейчас

- Не позиционируем как shipped **весь** хвост `/shop/b2b/`* (margin calculator, отчёты, десятки витрин) без одного выбранного демо-сценария.
- Не обещаем **единый enterprise realtime-чат** с одной серверной моделью треда на все роли без отдельной фазы/ADR.
- Не смешиваем **несколько семантик календаря** (brand vs shop delivery vs factory capacity) в один нарратив без продуктового решения «канонический календарь».
- Не тащим в P0 **Academy**, **client sewing patterns**, **админские метрики досье** как главный инвесторский экран.
- Не скрываем **демо vs prod**: organization hub, часть календаря, W2 localStorage — честная подпись в демо.
- Не раздуваем питч **Python FastAPI** без явной связки с тем, что видит пользователь в Next.
- Не требуем от каждого **showroom/linesheet** экрана отдельного e2e до сужения IA.
- Не перегружаем демо **планированием с AI-симуляцией** (`/brand/planning`) без явной связки «план ↔ заказ/артикул».

## Инвестору: что уже показываем как «работает» и чего не хватает

**Уже достаточно для живого прохода (при зелёных смоках и runbook):** три столба по URL (`**PLAN` §7.7**), W2 hub → артикул → секции/signoff, витрина `**/brand/showroom`**, реестр и карточка operational order, интеграции как карта, messages + calendar, честные подписи demo/LS/env (`**docs/INVESTOR_DEMO_RUNBOOK.md**`).

**Не хватает именно как «функция-история»** (продукт/код — см. `**GAP` §2, §5, §7.4–7.6**; визуально `**FINAL` must** п.2–3, 8):


| Пробел                                         | Почему инвестор не видит «как работает» end-to-end                             | Что закрывает                                                              |
| ---------------------------------------------- | ------------------------------------------------------------------------------ | -------------------------------------------------------------------------- |
| **Коллекция = только после финального сэмпла** | Нет единого правила в UI/API, что в сезоне только eligible                     | ADR №1 `GAP` §7.6 + экран/фильтр коллекций                                 |
| **OO → свод спроса → PO**                      | Агрегаты в коде есть, **сквозной** нарратив в UI и одном e2e — слабее обещание | ADR №2 + UI «merge / свод» + смоки Фазы 2 `PLAN` §3.1                      |
| **W2 «как в прод»**                            | Досье в демо часто LS — не доказательство персистентности                      | Персистентный путь + подпись в UI (`**FINAL` must** п.6)                   |
| **Один enterprise-чат**                        | Сознательно два UX — нужно объяснять, не показывать «одну кнопку»              | Остаётся нарратив + опционально ADR №3                                     |
| **Tech pack как shipped**                      | Без env только пилот                                                           | Preflight + честная фраза (**runbook** §3)                                 |
| **«Ролик / one-pager»**                        | Не входит в четыре research-файла                                              | Loom/PDF по `**INVESTOR_DEMO_RUNBOOK`** — отдельный маркетинговый артефакт |


**Вывод:** четыре файла + runbook **закрывают сценарий показа**; чтобы инвестор **сам** сказал «понял, как система работает в бою», нужны строки таблицы выше — это **внедрение в коде и ADR**, не расширение FOCUS/GAP/PLAN/FINAL бесконечно.
